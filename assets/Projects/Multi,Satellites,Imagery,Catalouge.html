While doing my MS in Remote Sensing & GIS, I needed to find some satellite images from multiple sensors around the world and work on them. I had to surf through multiple satellite sensor portals like USGS, Sentinel Hub, AtlasOne etc. and go through steps like signing up, account activations, drawing and downloading data from them before being able to use them on my end. This effort was hectic and ofcourse not much automated.


I accepted the challenge to design an online portal, which would house data from multiple satellite sensors under one room and would serve user with the best available dataset from multiple soruces. It involved multiple satellites, some of them were hosted locally on a custom built spatial database, others were interacted on the fly from their open-ended APIs.


Metadata parsers were written for locally hosted satellite sensors in c#, inserted into a postgresql database and online APIs were interacted dynamically for the AOI drawn by the user and results from both the sources was merged into one before being served over a web based gis portal that was built using Openlayers & Jscript


User can alter the criteria & satellites to filter out the results before adding up them to the cart and generating an online PDF report after selection of the products before selecting if they would like the raw imagery to be downloaded or served dynamically over the portal itself on the next login via a WMS service


Only the data that intersects witht the AOI for the selected scene, is clipped via a python script and saved onto the folder of the user. If ftp method is selected, nothing extra happens. If WMS is selected, another python script runs in the back-end on the server and creates a store on the geoserver for serving the imagery on the user portal when the user logs in next time.


This project is of it's own kind and not much work has been done in the field before. Individual catalogs are present for the respective satellite sensors but none of the portal shows it combined under a single roof.
